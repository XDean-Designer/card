模板编辑锁定

存在有效持卡时模板全量只读；无有效持卡时可编辑

现状

  • 详情页提供「编辑」入口，可修改 Step1 基本信息、Step2 权益组合、Step3 用卡策略并保存回模板。
  • 模板一旦被用于办卡，实例账本（实付、剩余权益、到期日、策略快照）与模板字段强关联；若允许随意改模板，已持卡实例与 Q1–Q4 规则将口径不一致。

核心矛盾

  • 商家希望「改规则」vs 已持卡须按办卡时模板约定消费、延期、计绩、退卡。
  • 需区分「改模板给新客用」与「改已有会员的持卡合约」。

锁定判定

  • 可编辑：该模板 当前有效 = 0（无未退卡且卡级未过期的持卡）。
  • 不可编辑：该模板 当前有效 > 0。
  • 重新解锁：全部持卡已退卡，或全部持卡已因到期/退卡变为非有效后，模板恢复可编辑。
  • 不计入锁定：历史「办卡人数」、已退卡记录、已失效（未退卡）的持卡均不触发锁定；仅以当前有效数量为准。
与运营指标对齐：锁定条件与详情页「当前有效」统计一致（未退卡且卡级未过期）。

锁定范围 · 全量只读

  • Step1 基本信息、Step2 权益组合、Step3 用卡策略均不可修改。
  • 仍允许：查看详情、上下架、克隆模板、为会员办卡/延期/退卡（对已持卡实例操作，不改模板）。
  • 保存接口：若 validCardCount > 0，拒绝模板更新并返回业务错误码。

改规则替代路径 · 克隆

  • 需调整权益/有效期/策略时：使用克隆模板 → 编辑新模板 → 保存为新卡型。
  • 新会员办理新模板；老会员仍绑定旧模板实例，账本与规则不变。
  • 旧模板可继续保留上架（老客续办/补录）或商家自行下架，不强制下架旧模板。
  • 克隆产物记录 duplicatedFrom（可选），便于追溯来源，不影响实例计算。
编辑锁定决策 打开模板详情 ├─ 当前有效 = 0 → 「编辑」可用 → 进入 Step1–3 修改并保存 └─ 当前有效 > 0 → 「编辑」置灰 提示：已有持卡，不可编辑；请克隆后修改 可操作：克隆 / 下架 / 查看 / 已持卡管理

界面表现

  • 详情页保留「编辑」按钮；当前有效 > 0 时按钮置灰不可点。
  • 置灰态提示文案：「已有持卡，不可编辑；请克隆后修改」（点击或 hover 展示均可）。
  • 「克隆」入口保持可用，与编辑按钮并列或置于同一操作区。
与 Q1–Q4 衔接：模板锁定保证实例侧 卡到期日、权益结构、usagePolicy 在持卡期内不被后台篡改；计绩与退卡账本仍以办卡实例为准。

下架模板 · 可见性与操作

  • 列表页顶部「在售 / 已下架」分段切换；下架后从「在售」移除，进入「已下架」列表，不删除模板与运营数据。
  • 已下架列表：卡面可进详情；底部仅「克隆」（隐藏「办卡」）；彩色卡条内名称右侧展示「已下架」白底胶囊标签。
  • 已下架详情：运营数据、卡面信息只读展示；彩色顶栏右侧展示「已下架」态 +「克隆」（白底描边胶囊）；底部双按钮「重新上架 / 已持卡管理」。
  • 新客办卡：已下架模板不可新办；已持卡仍可退卡、延期（实例操作，不改模板字段）。
  • 重新上架：恢复至「在售」列表，新客可再次办理；已持卡实例不受影响。
  • 克隆：下架卡可克隆为新模板（记录 duplicatedFrom），新卡默认上架。
演示数据：「有数据」注入 10 张卡；其中「儿童10次剪发」为已下架样例；另含永久储值、减值策略 / 减值烫染等对照卡。

问题 1 · 卡有效期与项目/折扣有效期如何闭环？

卡级有效期仅约束项目与折扣；面值永久有效

现状

  • 卡模板勾选「面值 / 项目 / 折扣」组合权益，模板层无卡级有效期。
  • 每个项目权益可单独设 有效期限（永久 / 6月 / 1年 / 自定义）。
  • 每个折扣权益可独立设「自办卡日起 X 有效」。
  • 导致多层到期日并存，消费/延期/退卡/业绩口径难以统一。

核心矛盾

  • 项目/折扣各自有效期 vs 卡整体「是否有效」语义冲突。
  • 面值余额无到期规则，与有次卡有效期的卡型混在一起时，失效边界不清。
  • 延期、退卡、劳动业绩均需要单一到期判定作为前提。

仅卡级有效期

卡级有效期统一管理全部权益

模板新增卡有效期（永久 / 预设 / 自定义），仅约束项目与折扣；删除项目权益、折扣权益的独立有效期配置。办卡时写入实例 卡到期日，项目/折扣随卡到期失效；面值永久有效。

  • 模板层：Step 权益配置中，项目仅保留次数/不限次，折扣仅保留折扣规则；不再出现「有效期限」行。
  • 办卡实例：仅持久化 卡到期日（永久卡为空或哨兵值）；不再为单项权益写 权益到期日。
  • 面值余额永久有效，不受卡级到期约束；卡到期后仍可消费/退面值。
  • 项目权益、折扣权益随卡级到期失效；到期当日硬失效，不设宽限期。
  • 消费校验：面值消费不校验卡到期；项目/折扣须卡未过期（或永久）且次数/规则适用。
  • 补录老卡：补录流程必填「原办卡日」；卡到期日 = 原办卡日 + 当前模板卡有效期（永久模板则到期为空）。不以补录操作日代替原办卡日。
  • 到期后处理：到期当日自动转「失效」，权益冻结，不设宽限期；恢复使用须通过 Q2「延期」或重新办卡；已失效卡允许延期复活（见 Q2）。
层级关系 会员卡实例 ├─ 卡到期日（仅约束项目 / 折扣） ├─ 面值权益（余额 + 赠送余额 · 永久有效） ├─ 项目权益[]（仅次数 / 不限次，随卡到期） └─ 折扣权益[]（仅折扣规则，随卡到期） 卡到期 → 项目/折扣不可用；面值仍可消费与退卡
闭环要点：「当前有效」统计仍看是否退卡；含项目/折扣时另看卡级是否过期。面值区展示「永久有效」，期限行仅展示给项目/折扣。

补充规则

补录老卡 · 按原办卡日推算

补录时必填原办卡日（不可仅用补录当天）。系统按当前模板卡有效期计算：

卡到期日 = 原办卡日 + 模板有效期（永久模板则 卡到期日 为空）。

尊重老卡真实持卡时长；避免补录日「洗白」延长有效期。

依赖店员知晓/核对原办卡日；补录页需增加日期字段与校验。

到期当日 · 硬失效（无宽限）

卡级到期日当天 0 点（或营业日规则）起，项目次数与折扣不可核销；面值余额仍可消费。

不设宽限期；到期前可通过消息提醒，但不延长可用期。

规则简单、与「当前有效」统计一致；减少到期后争议消费。

到期日当天仍想消费的客人须先延期；对忘记续期的体验较硬。

与 Q2 衔接：失效后若允许「延期复活」，则在 Q2 单独约定；本规则不自动给予宽限消费期。

用卡策略 · Step3 配置项

创建流程第三步必填；消耗退卡与业绩结算独立配置

字段与流程

  • 创建流程：Step1 基本信息 → Step2 权益组合 → Step3 用卡策略（必填）→ 保存。
  • 数据字段：usagePolicy: { refundRule, performanceRule }，各为 alloc_pool（均值，默认）或 retail（减值；代码枚举名仍为 retail）。
  • 独立性：两项可任意组合，共 4 种策略；新建卡默认 alloc_pool / alloc_pool。
  • 演示卡（有数据 · 10 张）：
    • 均值默认：尊享组合、新客首开、老客续充、儿童10次（已下架）、烫染会员价、全年洗吹、永久储值、可编辑演示（无持卡）。
    • retail / retail：减值策略演示卡（纯次数）、减值烫染卡（纯折扣，对照烫染会员价）。

消耗及退卡规则（refundRule）

alloc_pool · 均值模式（默认）Q4 对齐

办卡实付在统一名义价值池中一次分摊；余额/项目按分摊折算，折扣按累计已享优惠计消耗价值；建议退款 = max(0, 办卡实付 − Σ消耗价值)。

retail · 减值模式

项目：消耗价值 = 价目表挂牌价 × 已用次数；剩余 = 挂牌价 × 剩余次数。

余额：与均值相同，按实耗面值折算实付（充值优先消耗）。

折扣：消耗价值 = 已省优惠合计 Σ(原价−实付)；组合卡每线剩余 = 该线分摊实付 − 已省优惠（账本 savedTotal）。

建议退款：同均值 max(0, 办卡实付 − Σ消耗价值)。

业绩结算规则（performanceRule）

alloc_pool · 均值模式（默认）Q3 对齐

员工业绩按 Q3：余额/次数合并计池分摊，折扣按折后实收。

retail · 减值模式

不论支付来源，员工业绩均按当次服务项目价目表挂牌价计（原型 Step3 展示说明，核销计绩待正式对接）。

与 Q1 衔接：面值永久有效不影响策略配置；卡到期后项目/折扣失效，但面值仍可退卡估值（按所选 refundRule）。

Step3 界面 · 说明方式

  • 无 ⓘ 弹窗：计算方法直接写在各选项下方的条目副标题（小圆点分条）。
  • 界面名称统一为「均值模式 / 减值模式」；代码仍用 alloc_pool / retail。
  • 副标题与原型退卡引擎口径一致（见 Q4）。
均值模式 · 消耗及退卡（界面摘要）

· 消耗价值 = 消耗面值 ×（办卡实付 ÷ 初始面值）

· 纯折扣卡按累计已享优惠

· 建议退款 = 办卡实付 − 消耗价值

减值模式 · 消耗及退卡（界面摘要）

· 项目、折扣按挂牌价计消耗价值；有面值时余额按面值折算办卡实付

· 纯折扣卡按累计已享优惠

均值模式 · 业绩结算（界面摘要）

· 员工业绩 = 扣款额 ×（办卡实付 ÷ 初始面值）；次数按购卡分摊到每次；折扣按折后实收

减值模式 · 业绩结算（界面摘要）

· 不论余额 / 次数 / 折扣，员工业绩均按当次服务挂牌价计

问题 2 · 延期延什么？规则如何设置？

在 Q1 前提下，延期仅延长卡级有效期

现状

  • 选择会员弹窗 Tab：「办卡」/「退卡/延期」。在「退卡/延期」中对已持卡提供行内延期：时长控件（日 / 月 / 年 / 永久）+ 数值；确认后写入 卡到期日，失效卡可复活。
  • 列表卡面主按钮文案为「办卡/退卡」，进入同一选择会员弹窗。
  • 含「折扣」权益的卡（如烫染会员价卡、尊享组合卡、减值烫染卡）：延期面板展示延期费用（单位：元），默认按折扣权益分摊购卡价 × 延期天数 ÷ 原有效期折算（纯折扣卡等价于整卡购卡价）；商家可改价或填 0 免费；写入独立延期流水，不计入办卡总额。
  • 仅含面值或项目次数、无折扣的卡（如新客首开、儿童次卡、全年洗吹、永久储值）：延期免费，不产生延期订单；纯面值且卡永久有效时不展示延期入口。

结论（由 Q1 推导）

  • 权益项已无独立有效期，延长卡有效期 = 延长全部权益可用期，无需再选延期范围。
  • 延期操作对象唯一：卡到期日（+N 天 / 月 / 年，或延为永久）。
  • 卡模板为永久有效时，延期入口仅隐藏（不展示「无需延期」提示文案）。

仅延长卡有效期

  • 操作语义：时长控件确认天数 → 卡到期日 += N（或延为永久则清到期日）。
  • 权益联动：项目次数、折扣规则随卡续期恢复；面值本就永久有效，无需续期。
  • 界面简化：删除「选择延期权益」等多选步骤；提示文案改为「延长本卡有效期 N 天」。
  • 流水记录：写入延期流水（操作人、原到期日、新到期日、+N、是否收费）。
  • 与办卡区分：延期不等于续费购卡；收费延期走独立「延期订单」，不计入办卡总额/办卡次数（见下方补充规则）。

补充规则

延期费用 · 无折扣免费，含折扣卡收费

无「折扣」权益（仅面值 / 仅项目次数）：延期免费，不产生订单。

含「折扣」权益的卡（含纯折扣卡与组合卡）：延期时展示费用字段，默认金额为折扣权益分摊购卡价 × 延期天数 ÷ 原有效期（与 Q4 名义价值池折扣权重一致）；金额由商家自行填写，可填 0 免费；生成独立「延期订单/流水」，不计入办卡总额、办卡次数。

折扣权益无次数可消耗，延期实质为续享会员价，收费合理；组合卡仅对折扣部分折算，避免与余额/次卡重复计费。

原型已支持含折扣卡延期收费；需在正式环境对接延期订单与支付。

已失效卡 · 允许延期复活

卡已因到期转「失效」后，仍可通过延期操作：卡到期日 += N，状态恢复为有效，权益随卡续期重新可用（与 Q1 硬失效不矛盾，失效后靠延期「复活」）。

避免客人必须作废重办；与「退卡/延期」Tab 行内延期入口一致。

需在 UI 区分「即将到期」与「已失效」文案，避免误以为宽限期可消费。

问题 3 · 如何计算会员卡消费造成的店员劳动业绩？

均值模式 / 减值模式（performanceRule）

现状

  • 详情运营数据展示：办卡总额、持卡余额、项目消费次数等门店经营指标。
  • 消费核销、服务单、店员分成未在原型中建模。
  • 同一笔服务可能走：余额支付、次卡抵扣、折扣价支付，价值认定不同。

核心矛盾

  • 劳动业绩通常按服务「产值」计，但卡消费未产生当期现金流水。
  • 组合卡一次办卡多项权益，需避免「一次消费重复计绩」或「计绩为 0」。
  • 计绩口径需与 Q4 退卡估值共用同一套「权益账面价值」账本。

均值模式 · 按权益类型分计

余额 / 次数 / 折扣分计 · 赠送与购买合并计池默认 · alloc_pool
  • 余额消费：充值余额与赠送余额合并计算；业绩 = 扣款额 ×（办卡实付 / 办卡总面值），总面值 = 充值 + 赠送。
  • 次数消费：购买次数与赠送次数合并计池；单次摊销 = 该项目分摊实付 /（购买次数 + 赠送次数）；无论扣减购买次或赠送次，业绩均 = 单次摊销 × 本次次数。
  • 折扣消费：业绩 = 折后实收金额。
  • 组合卡分摊：办卡实付按各权益名义价值在统一分摊池中一次分配（见 Q4 均值模式）。纯项目卡时，实付即按「价目表原价 × 配置次数」比例分摊到各项目；含面值/折扣时，余额总面值、各项目名义价值、各折扣线名义权重共同参与分母，避免重复计价。
  • 计绩触发点：核销成功时写入 劳动业绩金额；账本字段与 Q4 退卡共用。
支付来源劳动业绩公式要点
余额（充值 + 赠送合并）扣款额 × 办卡实付 / (充值+赠送)赠送余额消耗计绩低于等额充值，口径统一
项目次数（购买 + 赠送合并）(分摊实付 / (购次+赠次)) × 本次次数赠送次与购买次同额计绩
折扣权益折后实收金额按当次服务实收计

减值模式 · 价目表原价计绩

不论支付来源，统一按挂牌价计绩retail

模板 usagePolicy.performanceRule = retail 时启用；覆盖均值模式上表全部分项公式。

  • 统一规则：单次服务劳动业绩 = 当次服务项目在价目表中的原价（与扣款额、办卡分摊、折后实收无关）。

补充规则

折扣权益 · 退卡按累计已享优惠

组合卡中折扣线仍纳入统一名义价值池（用于算该线分摊实付与剩余明细），但消耗价值按累计已享优惠计：

消耗价值 = Σ(原价 − 折后实付)（账本 savedTotal） · 剩余价值 = max(0, 分摊实付 − 消耗价值)

建议退款 = max(0, 办卡实付 − Σ消耗价值)，不以 Σ剩余封顶。

与 Step3「纯折扣卡按累计已享优惠」副标题一致；均值/减值折扣线口径统一。

需能核算每次折扣的已省金额（固定价或折率 × 价目表原价）。

问题 4 · 退卡时如何计算价值消耗与应退金额？

均值模式 / 减值模式 · 系统建议退款 + 人工确认

现状

  • 「退卡/延期」Tab 支持退卡估值 Sheet：展示办卡实付、消耗价值、权益剩余明细（右对齐；有消耗的行额外显示该项消耗价值）、系统建议退款、可编辑实际退款（单位：元）。
  • Sheet 副标题标注策略：· 均值退卡 / · 减值退卡。
  • 原型已支持退卡估值 Sheet 演示（含多场景持卡样本，见 10 张演示卡）。
  • 手续费规则、超额退款审批、退卡备注等尚未实现（人工确认环节部分能力）。
用语：界面与汇总字段统一使用「消耗价值」；建议退款 = max(0, 办卡实付 − Σ消耗价值)，与 Step3 副标题一致。

核心矛盾

  • 退卡需回答：消耗价值 + 剩余价值 / 建议退款 + 实际退款（可人工调整）。
  • 须与 Q3 业绩规则共用账本，避免「计绩一套、退卡另一套」。

均值模式 · 消耗价值算法

先算消耗价值，再建建议退款默认 · alloc_pool

退卡核心量是消耗价值（非「面值」本身）：建议退款由办卡实付 − 消耗价值得出，且不得超过办卡实付。店员可确认或调低实际退款。

产品口径（与 Step3 副标题一致）

场景消耗价值说明
有面值（含纯面值） 消耗价值 = 消耗面值 ×（办卡实付 ÷ 初始面值） 初始面值 = 充值 + 赠送；消费先扣充值面值再扣赠送
纯折扣 / 折扣线 消耗价值 = 累计已享优惠 与 Step3 副标题一致；减值模式同述
建议退款 建议退款 = 办卡实付 − 消耗价值 与 Step3 副标题一致；不得超过办卡实付
与引擎一致：组合卡项目次数仍按购卡分摊摊销；折扣线（均值/减值）均为累计已享优惠；建议退款均为 max(0, 办卡实付 − Σ消耗价值)。Sheet 仍可展示各线「剩余价值」作明细，但不参与封顶。

引擎实现 · 名义价值池（组合卡分摊）

多权益组合时，须先把办卡实付一次分到各权益线，再套用上表；否则会出现「各线各自用满实付」的重复计价。实现函数 buildRefundNominalPool / computeSuggestedRefundAllocPool：

步骤公式说明
① 名义价值池 余额名义 = 充值+赠送总面值
项目权重[i] = 价目表单价 × (购次+赠次)
折扣权重[k] = 价目表单价（或全部项目均价）
名义合计 = Σ 权重		 不限次项目不参与；权重为 0 的折扣线跳过
② 余额 · 消耗价值 余额分摊实付 = 办卡实付 × (余额名义 / 名义合计)
消耗价值 = 已用充值 × (余额分摊实付 / 充值面值) + 已用赠送 × (余额分摊实付 / 总面值)
剩余价值同理用剩余充值/赠送折算		 纯面值时名义合计=总面值，简化为「消耗面值 × 实付 ÷ 初始面值」口径
③ 项目 · 消耗价值 分摊实付 = 办卡实付 × (项目权重 / 名义合计)
单次摊销 = 分摊实付 / (购次+赠次)
消耗价值 = 已用次数 × 单次摊销		 购/赠同单价
④ 折扣 · 消耗价值 分摊实付 = 办卡实付 × (折扣权重 / 名义合计)
消耗价值 = 累计已享优惠 Σ(原价 − 折后实付)（账本 savedTotal）
剩余价值 = max(0, 分摊实付 − 消耗价值)		 与减值折扣口径相同；纯折扣卡时名义合计仅含折扣权重
⑤ 建议退款 建议退款 = max(0, 办卡实付 − Σ消耗价值) Sheet 汇总「消耗价值」= Σ 分项消耗价值；不以 Σ剩余封顶
  • 人工确认：店员可下调「实际退款」（单位：元）；高于建议需授权（原型未实现）。
  • 退卡后：持卡失效、权益清零、写退卡流水，联动「当前有效」等运营指标。
退卡估值流程（均值 · 原型） 1. 读取持卡账本（办卡实付、各权益剩余/已用 · 含赠送） 2. 构建名义价值池 → 名义合计 3. 分项计算 消耗价值 / 剩余价值 4. 建议退款 = max(0, 办卡实付 − Σ消耗价值) 5. Sheet：办卡实付 · 消耗价值 · 权益剩余明细（有消耗显示分项消耗，金额右对齐）· 建议退款 6. 店员确认/修改 实际退款（元） 7. 写退卡流水 + 更新运营指标
反例（已修复）：若余额按「实付/总面值」、项目又按「实付×项目比例」各自独立用满实付，组合卡会重复加计。组合卡必须共享同一名义合计分母。

减值模式 · 消耗价值算法

项目 / 折扣按挂牌价，余额同均值折算retail

模板 usagePolicy.refundRule = retail 时启用。建议退款同样：max(0, 办卡实付 − Σ消耗价值)。

权益消耗价值剩余价值说明
余额 与均值模式相同（先扣充值、再扣赠送；按分摊实付折算） 面值线不受减值规则替换
